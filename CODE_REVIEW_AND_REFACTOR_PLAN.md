## 1. Что сейчас представляет собой проект

Игра «Козёл»: 4 игрока, у каждого своя козырная масть (SPADES/CLUBS/DIAMONDS/HEARTS).
Колода 36 карт (9 рангов × 4 масти). Шестёрки сразу уходят на «доску счёта» (scoreboard),
затем 7-ки обязательно обмениваются владельцам своих мастей, игроки выбывают, когда
исчерпывается их козырная масть. Побеждает последний оставшийся.

### Текущая карта файлов (src/main/java/ru/anseranser)

| Файл | Строк | Роль | Проблемы |
|------|-------|------|----------|

## 2. Главные проблемы по категориям

### 2.1 Архитектура / разделение ответственности (критично для портирования)
1. **Логика слита с I/O.** `Game` и `Player` пишут в `System.out` прямо внутри правил игры
   (`setupRound`, `endRound`, `playLeadCard`, `beatCard`, `takeBank`). Для мобильного/настольного UI
   правила игры должны быть переиспользуемыми и НЕ зависеть от консоли.
2. **Ввод слит с моделью.** `HumanPlayer` содержит `Scanner` и интерактивный опрос. В UI ввод
   придёт с кнопок/тача — этот код не переиспользуется.
3. **Отсутствует понятие «состояние игры» / события.** Сейчас всё течёт через печать. Для UI
   нужен либо возврат снимка состояния после каждого шага, либо `GameListener`/`EventPublisher`.
4. **`Game.playGame()` прокручивает всю партию в один вызов** и печатает по ходу. UI должен
   продвигать игру пошагово (ход, раунд, партия) и рендерить между шагами.
5. **Нарушение инкапсуляции.** `Game` дёргает `loser.setGamer(false)` / `setRounder(false)` —
   управляет внутренним состоянием игрока. Жизненный цикл игрока должен жить в движке.

### 2.2 Дублирование логики (упрощение)
6. **`HumanPlayer.makeMove` почти дословно копирует `Player.makeMove`** (лид/битва/взятие банка).
   Любой багфикс в базовом методе нужно вручную повторять в наследнике — хрупко.
7. **Паттерн обхода круга скопирован ~15 раз** (`Player start = players.getRandom(); current = start;
   do { … } while (current != start)`) в `Game` и тестах. Это должен быть один хелпер/
   итератор (`forEachPlayer`, `streamPlayers`, `playersInOrderFrom`).
8. **Непоследовательная терминология:** «bank» / «pot» / «bank» используются как синонимы
   (`takeBank` vs `takePot` vs поле `bank`). Нужно один термин (в «Козле» — «банк»/«прикуп»).

### 2.3 Тестируемость / детерминизм
9. **`ThreadLocalRandom` в `getRandom()` и `Collections.shuffle(deck)`** делают партии
   недетерминированными. Для тестов и «replay»/сидов нужен инъектируемый `Random`.
10. **`CircularDoublyLinkedList` не реализует `Iterable`/`List`** — отсюда ручные циклы и
    возвраты `null`. Замена на стандартную коллекцию упростит код и уберёт O(n)-поиски.

### 2.4 i18n / локализация (критично для перевода)
11. **Все строки захардкожены на английском** внутри модели: `"Looser"`, `"can`t beat"`,
    `"THE GAME BEGINS"`, `"ROUND"`, и т.д. Опечатки: «Looser»→«Loser», «can`t»→«can't».
12. **`Card.toString()` возвращает `"SIX of SPADES"`** — это и данные, и представление сразу,
    и только на английском. Модель должна быть языково-нейтральной; отображение — задача view/форматтера.
13. **Масти и ранги не имеют локализуемых имён/символов** (♠♣♦♥, «шестёрка пик» и т.п.).

### 2.5 Потенциальные баги (выявлено при ревью)
14. **Мёртвый/опасный код в `HumanPlayer.chooseDefenseCard`:** ветка
    `choice == validDefenses.size() + 1` возвращает `null`, но в `makeMove` далее делается
    `bank.add(beatCard)` с `beatCard == null` → `NullPointerException`. Сама ветка по комментарию
    «не должна срабатывать», но защиты нет. Нужно либо убрать опцию «взять банк», либо корректно
    обработать её (передать в takePot).
15. **Старт раунда от дилера без проверки, что дилер — «rounder»:** `playMoves(current)` вызывает
    `current.makeMove(bank)` на дилере, даже если у дилера пустая рука (тогда `makeMove` сразу
    возвращает). Логика выживает, но хрупка и неочевидна.
16. **`endRound(loser, bank)` получает поле `bank` как аргумент** — избыточный параметр,
    семантика «возвращает true = выбыл» запутана именованием метода.
17. **`nextRequiredRank` + ACE-пропуск** реализуют правило «обмен 7→8→…→K, никогда тузы»,
    но это поведение не задокументировано и не покрыто отдельным тестом `Scoreboard`.

## 3. Рекомендуемая целевая архитектура

Принцип: **чистое, независимое от платформы ядро игры** + **тонкие платформенные UI/контроллеры**.

### 3.1 Предлагаемая структура пакетов (JVM, сохраняем Java/Kotlin-ядро)

```
ru.anseranser
├── core                      # ЧИСТОЕ ЯДРО: нет System.out, нет Scanner, нет UI
│   ├── model
│   │   ├── Card.java         # record(suit, rank) — нейтральные данные, БЕЗ toString-текста
│   │   ├── Suit.java         # enum с id (без строк отображения)
│   │   ├── Rank.java         # enum + value
│   │   ├── Player.java       # ТОЛЬКО состояние + стратегия выбора (MoveStrategy)
│   │   └── GameSnapshot.java # immutable снимок состояния (для UI / save-load)
│   ├── engine
│   │   ├── GameEngine.java   # оркестрация: шаги, раунды, выбывание (без I/O)
│   │   ├── Dealer.java       # сборка колоды, shuffle(с инъекцией Random), раздача
│   │   ├── Scoreboard.java   # suit→Deque + правила nextRequiredRank (покрыто тестами)
│   │   └── TurnOrder.java    # обход по кругу: nextGamer/nextRounder/forEach (замена CDLL)
│   ├── strategy
│   │   ├── MoveStrategy.java # интерфейс: chooseLead / chooseDefense
│   │   ├── AiMoveStrategy.java   # текущая эвристика Player
│   │   └── HumanMoveStrategy.java # запрашивает выбор через InputProvider (без Scanner)
│   └── rules
│       └── CardRules.java    # canBeat / compare / weakestDefense (вынесено из Player)
├── input
│   └── InputProvider.java    # контракт ввода (реализации: Console, UI, Bot)
├── i18n
│   ├── Messages.java         # обёртка над ResourceBundle по ключам
│   ├── CardLocalizer.java    # масть/ранг → локализованное имя/символ
│   ├── messages.properties        # ключи (английский / fallback)
│   ├── messages_ru.properties
│   └── messages_xx.properties
├── app
│   └── GameController.java   # связывает engine + input + выдаёт события/снимки UI
└── ui
    ├── console
    │   └── ConsoleView.java   # текущее поведение, переписанное поверх controller
    └── (будущие) desktop / android / ios
```

> Если целевая платформа включает НЕ-JVM (iOS), рассмотреть **Kotlin Multiplatform** для `core`
> ( `expect/actual` по платформам). Если только Android + Desktop (JVM) — обычный Java/Kotlin-модуль
> с `:core` (pure JVM) и `:app` (JavaFX/Compose/Android) достаточен. Решение зависит от цели —
> зафиксировать до начала переноса UI.

### 3.2 Ключевые принципы
- **Ядро не производит строк.** Оно производит события/снимки состояния. Отображение и перевод —
  задача слоя `ui` + `i18n`.
- **Шаговый драйв.** `GameEngine` предоставляет методы уровня шага: `startGame()`, `playRound()`,
  `stepMove()`, возвращающие `GameSnapshot`. UI вызывает их и рендерит результат.
- **Инъекция зависимостей.** `Random`, `InputProvider`, `Locale`, список игроков — через конструктор/конфиг.
- **Стратегия выбора карт — единая точка.** `Player.makeMove` становится ОДНИМ универсальным циклом,
  который вызывает `MoveStrategy`. ИИ и человек отличаются только источником выбора.

## 5. Что упрощается (итог для раздела «упростить логику»)

- Один универсальный `makeMove` вместо двух (Player + HumanPlayer) → −50 строк дубликата.
- Один `TurnOrder`/`forEach` вместо ~15 скопированных циклов обхода.
- `CircularDoublyLinkedList` (186 строк, O(n), null) → стандартная коллекция + хелпер.
- Чёткое разделение: ядро (правила) ↔ стратегия (ИИ/человек) ↔ ввод ↔ вывод ↔ перевод.
- Пошаговый драйв вместо «прокрутил всю игру и напечатал» → готово к любому UI.

## 6. Риски и что держать в голове

- **Сохранить поведение.** Рефакторинг НЕ должен менять правила. Опираться на `GameSimulator`
  и «золотой» вывод из Этапа 0.
- **Правило обмена 7→K.** Оно сейчас имплицитно в `nextRequiredRank` + ACE-пропуске. Перед
  декомпозицией задокументировать точное правило и покрыть тестом, иначе легко сломать.
- **Кто такой «dealer» и порядок ходов.** Убедиться, что после рефакторинга порядок ходов и
  старт от дилера совпадают с текущим (см. п. 2.5-15).
- **Сиды/детерминизм.** Инъекция `Random` — обязательна до написания UI (иначе невозможен replay
  и стабильные тесты сценариев).

> Дата выполнения: 2026-07-11. Поведение игры НЕ изменилось (9 тестов зелёные).

### Зачем
Единственный оставшийся источник недетерминизма в ядре — `Collections.shuffle(deck)`
с неявным `Random` по умолчанию. Без инъекции RNG нельзя писать стабильные тесты
на правила и детерминированно воспроизводить партию (нужно для UI-портов и отладки).
После Этапа 3 (TurnOrder) и Этапа 2 (хуки) вся остальная логика уже детерминирована,
поэтому инъекция shuffle — последний шаг к полной воспроизводимости.

### Что сделано
- `model/Game.java`:
  * добавлено поле `@Setter private Random rng = new Random();`;
  * `shuffleAndDeal()` теперь делегирует `shuffleAndDeal(Random)` с инъектированным
    `rng` (использует `Collections.shuffle(deck, rng)` вместо дефолтного);
  * добавлен `playGame(Random rng)` — устанавливает rng и запускает игру; `playGame()`
    (без аргументов) идёт со стандартным `new Random()`.
- `GameTest.java`: добавлены два теста детерминизма:
  * `determinism_sameSeed_sameGame` — две игры с `new Random(42)` дают одного и того же
    победителя (по месту/trump) и побайтово одинаковые руки на каждом месте;
  * `determinism_differentSeed_canDiffer` — разные сиды обрабатываются без ошибок
    (проверка, что скрытая недетерминированность не вернулась).

### Примечания / подводные камни (чтобы не повторять)
- `Player` НЕ переопределяет `equals`, поэтому сравнивать `getWinner()` по объекту
  между двумя `Game` нельзя (разные инстансы = разные объекты даже для одного места).
  В тесте сравниваем победителя по `getTrump()` (месту), а руки — поэлементно
  (`Card` — record, value-based equals, сравнение списков корректно).
- Добавление перегрузки `playGame(Random)` сделало `game::playGame` неоднозначным в
  `HumanPlayerStage1Test` (метод-ссылка матчится с обеими перегрузками). Исправлено

## 14. Журнал фактически выполненных изменений (Stage 7 — симулятор и верификация)

### Что добавлено

  собирает `Result` (seed, победитель, totalCards, distinctCards, activeGamers).
  Инвариант `invariantsHold()` = 36 карт всего + 36 distinct + ровно 1 gamer.
- `GameSimulatorTest.java` (новый, 4 теста):

  * `determinism_sameSeed_sameWinner` — 50 сидов: повторный прогон даёт того же

  * `winners_areDistributedAcrossSuits` — 500 сидов: каждая из 4 мастей выигрывает

### ПОЙМАННЫЙ БАГ: бесконечный цикл раунда (ДО рефакторинга не ловился)
Симулятор на 750+ сидах НЕ завершался (вис на первом раунде, 2.2M «ходов»,
0 завершений раунда для части сидов; seed=0 — эталонный). БАГ ПРЕДШЕСТВУЮЩИЙ:

одну партию со случайным сидом, которая случайно завершалась, поэтому баг был
латентным.

Первоначальная гипотеза («взявший должен выйти из раунда → `rounder=false`») была
ОШИБОЧНОЙ: пользователь уточнил правило — взявший банк НЕ выбывает из текущего
раунда, выбывает только когда у него кончились карты на руках. Этот «фикс» сломал
бы правило, поэтому откатан.

КОРЕНЬ (доказан трассировкой событий): при правиле «взявший остаётся, выбывание
только при пустой руке» и ПРИ ОТСУТСТВИИ колоды-прикупа (stock) зацикливание
КОНЦЕПТУАЛЬНО НЕИЗБЕЖНО для части раскладов. Трасса seed=0:
```
P:DIAMONDS=SEVEN of SPADES   DIAMONDS ходит козырем SPADES
B:SPADES=JACK of SPADES/vsSEVEN of SPADES   SPADES бьёт старшим SPADES
P:SPADES=JACK of DIAMONDS     SPADES ходит козырем DIAMONDS
B:DIAMONDS=ACE of DIAMONDS/vsJACK of DIAMONDS  DIAMONDS бьёт старшим DIAMONDS
... (карты гоняют рука↔банк) ...
T:DIAMONDS(pot=9)             DIAMONDS берёт банк (рука РАСТЁТ)
... цикл повторяется ...
```
Двое игроков вечно обмениваются козырями: взятие лишь возвращает карты в руку,
никто не выбывает, `countActiveRounders()` никогда не падает до ≤1. В классическом
«Козёл»/«Дурак» завершение гарантирует именно прикуп (stock): после отбоя защитник
добирает карты, и когда прикуп пустеет, руки малы и кто-то не может отбиться.
БЕЗ прикупа при данном правиле зацикливание неизбежно.

### Исправление (НЕ ломает правило пользователя — защитный guard, не смена правил)
- `Player.makeMove`: оставлен РОВНО по правилу пользователя — взявший банк
  `return`, `rounder` не сбрасывается (остаётся в раунде). Изменение откатано.
- `Game`: добавлен `MAX_ROUND_MOVES = 10_000` — жёсткая верхняя граница числа
  ходов в раунде. `playMoves` возвращает счётчик; если граница достигнута при
  `countActiveRounders() > 1` (патологический незавершающийся расклад), раунд
  принудительно завершается, а проигравшим объявляется rounder с НАИБОЛЬШИМ
  числом карт в руке (`mostCardsRounder()`) — естественный исход «последний игрок
  с картами проигрывает». Это страховка, а не смена правил: на нормально
  завершающихся раскладах guard НИКОГДА не срабатывает (нормальный раунд — десятки
  ходов). Проверено на 3 сидах вручную: seed=0 теперь завершается via guard
  (18 230 событий), seed=1/2 — естественно.
- `Game.allCards()` (новый публичный метод): сумма карт во всех руках + стеках
  scoreboard + пуле колоды дилера. Нужен симулятору для проверки сохранности.

### ВАЖНО: уточнение инварианта сохранности карт
Изначально в тесте считал только карты в РУКАХ — падало (было 8 из 36). На самом
деле «в игре» карты живут в трёх местах: руки + стеки scoreboard (выложенные
«шестёрки»…«тузы» лестниц) + пул колоды дилера (остаток после последнего раунда).
Корректный инвариант сохранности = руки + scoreboard + dealer.deck == 36. Именно
его и проверяет симулятор через `Game.allCards()`.
