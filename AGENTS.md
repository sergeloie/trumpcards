# AGENTS.md — Правила игры и архитектура

## Правила игры

### Колода и правила раздачи
- Игра называется "Свои козыри"
- Игра ведётся обычной карточной колодой в 36 карт
- Играют 4 игрока
- Перед каждой сдачей карт игрокам колода тасуется
- Карты раздаются по очереди, по одной, каждому игроку, по часовой стрелке, начиная с игрока, следующего после дилера, раздаются все карты имеющиеся в игровой колоде
- Перед самой первой сдачей карт дилер выбирается случайно, при следущих раздачах дилером становится игрок проигравший на предыдущем раунде игры

### Ход игры
- Каждому из игроков назначается своя масть в качестве козырной
- Из изначальной колоды извлекаются все карты достоинстов "6" каждой масти и выкладываются отдельно, это будет нужно в дальнейшем для отслживания хода игры. Место, где лежат эти карты будем именовать "Табло"
- Раздаются карты, по правилам, определённым выше
- Если у кого-то из игроков есть чужая козырная карта наименьшего достоинства (для первого раунда это семёрка), то она передаются тому игроку, кому эта масть назначена в качестве козырной. Из этого правила есть одно исключение, если у игрока из его масти остался только туз, то туз не подлежит такой передаче в начале раунда.
- Первый ход совершает дилер. Он кладёт на игровой стол одну карту. Игрок, следующий ща ним по часовой стрелке, должен побить эту карту и положить сверху ещё одну. Побить карту можно картой той же масти, но большего достоинства или своим козырем.
- Если у игрока есть возможность побить карту, положенную перед ним на игровой стол, он обязан побить её, поддаваться не дорускается.
- Если игрок никак не может побить карту, то он забирает все карты, ранее выложенные на стол в этом раунде. Если остались ещё игроки с картами, то следующий игрок снова ходит с одной карты.
- Таким манером игровой раунд продолжается, пока не останется только один игрок с картами на руках. он является проигравшим в это раунде.
- Из игровой колоды извлекается карта наименьшего достоинства, той масти, которая назначена проигравшему раунд игроку в качестве козырной и кладётся на Табло поверх карт той же масти, что там уже лежат
- Таким образом проигравший игрок наказывается, уменьшением количества его козырей в колоде.
- Глядя на табло, другие игроки могут определять, сколько у оппонентов козырей и какую наименьшу карту чужой масти им отдавать в начале раунда.
- Когда игрок лишается всех своих козырей, он выбывает из игры. Игру продолжают оставшиеся игроки, пока не останется один победитель.

Техническое задание для кодинг-агента

## 1. Общая цель

Разработать карточную игру в виде web-приложения по указанным выше правилам с:

* Backend: Java 21 + Spring Boot 4.1.0, Gradle - Kotlin
* Frontend: TypeScript + HTML5 Canvas
* Архитектура: Clean Architecture (Hexagonal / Ports & Adapters)

Игра должна:

* запускаться локально (single-player / hot-seat)
* игровое поле адаптивное и занимает всё окно браузера
* быть изначально спроектирована для последующего перехода в сетевой multiplayer (WebSocket / REST)

---

## 2. Архитектурные принципы

### 2.1 Основные правила

1. Чистая архитектура (строго):

   * Domain НЕ зависит ни от чего
   * Application зависит только от Domain
   * Infrastructure зависит от Application и Domain

2. Никакой логики игры вне Domain слоя

3. UI (Canvas) — только отображение + ввод

4. Backend не знает про Canvas

---

## 4. Domain слой (главное ядро)

### 4.1 Обязанности

* правила игры
* состояние игры
* валидация ходов
* переходы состояний

### 4.2 Основные сущности

Примеры (адаптируй под свои правила):

* Game
* Player
* Card
* Deck
* Turn
* Action

### 4.3 Правила

* Domain должен быть **чистым Java (без Spring)**
* immutable где возможно
* никакого IO

---

## 5. Application слой

### 5.1 Use Cases

Каждое действие — отдельный use case:

* CreateGame
* JoinGame (для будущего)
* StartGame
* PlayCard
* EndTurn

### 5.2 Ports

#### In (входящие)

```
interface PlayCardUseCase {
    GameState handle(PlayCardCommand command);
}
```

#### Out (исходящие)

* GameRepository
* EventPublisher (для будущего realtime)

Обязательно:

1. GameId как ключ

5. Поддержка событийной модели
